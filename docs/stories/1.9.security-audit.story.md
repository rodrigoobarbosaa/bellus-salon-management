**I want** que o aplicativo seja seguro contra vulnerabilidades comuns,
**so that** os dados dos meus clientes e do negócio estejam protegidos.

## Acceptance Criteria

1. Todas as server actions validam autenticação (user != null) antes de qualquer operação
2. Todas as queries incluem filtro `salao_id` (isolamento multi-tenant)
3. Nenhuma API route pública expõe dados sem autenticação (exceto `/api/booking/slots`, `/api/opt-out`, `/booking/[slug]`)
4. Inputs de formulário validados: email format, string length limits, numeric ranges
5. Nenhum segredo (.env keys) hardcoded no código fonte
6. Headers de segurança configurados: X-Frame-Options, X-Content-Type-Options, Referrer-Policy
7. Rate limiting básico em endpoints públicos (`/api/booking/slots`)
8. Server actions de equipe.ts validam enum `role` antes de cast
9. `updateProfissionalServicos` verifica que serviços pertencem ao mesmo salão
10. `npm run build` sem erros

## Dev Notes

### Padrão de Validação de Input

```typescript
// Validação de email
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
if (!emailRegex.test(email)) {
  return { error: "Email inválido." };
}

// Validação de enum
const validRoles = ["proprietario", "profissional"] as const;
if (!validRoles.includes(role as typeof validRoles[number])) {
  return { error: "Cargo inválido." };
}

// Validação de comprimento
if (nome.length > 100) {
  return { error: "Nome muito longo (máximo 100 caracteres)." };
}
```

### Rate Limiting In-Memory

```typescript
const rateLimit = new Map<string, { count: number; resetAt: number }>();

function checkRateLimit(ip: string, limit = 30, windowMs = 60000): boolean {
  const now = Date.now();
  const entry = rateLimit.get(ip);
  if (!entry || now > entry.resetAt) {
    rateLimit.set(ip, { count: 1, resetAt: now + windowMs });
    return true;
  }
  entry.count++;
  return entry.count <= limit;
}
```
