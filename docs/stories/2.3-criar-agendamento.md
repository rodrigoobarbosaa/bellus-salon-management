## Dev Notes

### ⚠️ Regra de Conflito Dual (Profissional vs Cliente)

**REGRA DE NEGÓCIO IMPORTANTE:**

| Quem agenda | Conflito detectado | Comportamento |
|-------------|-------------------|---------------|
| **Profissional** (dashboard) | Horário sobreposto | ⚠️ Mostra confirmação: "Já existe agendamento com [cliente X] neste horário. Deseja agendar mesmo assim?" — **PERMITE** continuar |
| **Cliente** (portal público) | Horário sobreposto | 🚫 **BLOQUEIA** — slot não aparece como disponível. Cliente nunca pode agendar no mesmo horário que outra |

**Motivo:** Na prática do salão, profissionais às vezes precisam encaixar duas clientes simultaneamente (ex: uma processando cor enquanto atende outra). Mas no portal público, isso não pode acontecer para evitar conflitos de expectativa.

### Conflito de horário — Query
```sql
SELECT id, cliente_id FROM agendamentos
WHERE profissional_id = $1
  AND status NOT IN ('cancelado')
  AND data_hora_inicio < $new_fim
  AND data_hora_fim > $new_inicio
```
- **Dashboard (profissional):** Se query retorna resultados → mostrar modal de confirmação com detalhes do conflito → permitir prosseguir se confirmado
- **Portal público (cliente):** Se query retorna resultados → bloquear slot, não exibir como disponível

### Cálculo de fim
`data_hora_fim = data_hora_inicio + servicos.duracao_minutos`

### Preço
```
preco_efetivo = servicos_profissionais.preco_override ?? servicos.preco_base
```

### Padrões
- Modal/Dialog usando componente `Dialog` do projeto
- Server Action para criar agendamento com validação server-side
- Busca de clientes com debounce (Client Component)
- Formato de hora: 24h (Europa)
