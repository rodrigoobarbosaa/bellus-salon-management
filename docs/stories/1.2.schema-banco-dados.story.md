# Story 1.2: Schema do Banco de Dados

## Story

**As a** data engineer do Bellus,
**I want** criar o schema completo do banco de dados PostgreSQL no Supabase,
**so that** todos os módulos do sistema (agenda, clientes, caixa, fiscal, lembretes) tenham uma base de dados normalizada, segura e com RLS configurado.

---

## Acceptance Criteria

1. Tabela `profissionais` criada (id, nome, email, telefone, role, cor_agenda, ativo, created_at)
2. Tabela `clientes` criada (id, nome, email, telefone, idioma_preferido, notas, intervalo_retorno_dias, created_at)
3. Tabela `servicos` criada (id, nome, descricao, duracao_minutos, preco_base, categoria, intervalo_retorno_dias, ativo)
4. Tabela `servicos_profissionais` criada — relação N:N com preço override por profissional
5. Tabela `agendamentos` criada (id, cliente_id, profissional_id, servico_id, data_hora_inicio, data_hora_fim, status, notas, created_at)
6. Tabela `pagamentos` criada (id, agendamento_id, valor_bruto, desconto, valor_liquido, iva_pct, iva_valor, forma_pagamento, created_at)
7. Tabela `transacoes_fiscais` criada (id, pagamento_id, trimestre, ano, tipo [IVA/IRPF], valor, created_at)
8. Tabela `lembretes_config` criada (id, servico_id, cliente_id NULLABLE, intervalo_dias, template_mensagem, ativo)
9. Tabela `notificacoes_log` criada (id, cliente_id, agendamento_id NULLABLE, tipo, mensagem, status, enviado_em)
10. RLS (Row Level Security) habilitado em todas as tabelas
11. Políticas RLS: profissionais só veem dados do próprio salão; clientes sem acesso direto
12. Indexes criados para campos de busca frequente (agendamentos.data_hora_inicio, clientes.telefone)
13. Migration SQL gerada em `supabase/migrations/` com timestamp
14. Seed de dados para desenvolvimento (`supabase/seed.sql`) com 2 profissionais, 5 serviços e 3 clientes

## Dev Notes

### Diagrama de Relações (ERD simplificado)

```
auth.users (Supabase Auth)
    │
    └── profissionais (1:1)
            │
            ├── servicos_profissionais (N:N) ←── servicos
            │
            └── agendamentos
                    │
                    ├── clientes
                    │
                    └── pagamentos
                            │
                            └── transacoes_fiscais

lembretes_config ──── servicos (N:1)
                 ──── clientes (N:1, opcional)

notificacoes_log ─── clientes (N:1)
                 ─── agendamentos (N:1, opcional)
```

### Convenções SQL do Projeto

- **IDs:** `uuid` gerado com `gen_random_uuid()` — PRIMARY KEY DEFAULT
- **Timestamps:** `created_at TIMESTAMPTZ DEFAULT NOW()`, `updated_at TIMESTAMPTZ`
- **Soft delete:** campo `ativo BOOLEAN DEFAULT true` (não deletar registros)
- **Nomes:** snake_case em português (ex: `data_hora_inicio`)
- **Moeda:** `DECIMAL(10,2)` para valores monetários (em EUR €)

### Exemplo de Tabela Completa

```sql
CREATE TABLE clientes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  nome VARCHAR(255) NOT NULL,
  email VARCHAR(255),
  telefone VARCHAR(20) NOT NULL,
  idioma_preferido VARCHAR(2) DEFAULT 'es' CHECK (idioma_preferido IN ('pt', 'es', 'en', 'ru')),
  notas TEXT,
  intervalo_retorno_dias INT,  -- NULL = usa o padrão do serviço
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE clientes ENABLE ROW LEVEL SECURITY;
```

### Política RLS Padrão

```sql
-- Profissionais autenticados veem todos os clientes do salão
CREATE POLICY "profissionais_podem_ver_clientes"
ON clientes FOR SELECT
TO authenticated
USING (true);  -- Por ora, todos profissionais do salão compartilham acesso

-- Somente service_role pode inserir/atualizar
CREATE POLICY "service_role_full_access"
ON clientes FOR ALL
TO service_role
USING (true);
```

```bash
# Aplicar migrations

# Gerar types TypeScript do schema
```
