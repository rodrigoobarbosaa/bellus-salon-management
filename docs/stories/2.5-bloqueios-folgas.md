## Story

Como **proprietário**, quero **bloquear horários e registrar folgas** para profissionais, para que o sistema não permita agendamentos nesses períodos e a agenda reflita a disponibilidade real.

---

## Acceptance Criteria

- [ ] AC-01: Proprietário pode criar bloqueio para um profissional (data/hora início e fim)
- [ ] AC-02: Proprietário pode criar folga de dia inteiro para um profissional
- [ ] AC-03: Bloqueios aparecem no calendário com visual distinto (cor cinza, hachurado)
- [ ] AC-04: Sistema impede criar agendamento em horário bloqueado

- [ ] AC-06: Campo de motivo opcional (ex: "Almuerzo", "Vacaciones", "Curso")

## Dev Notes

### Nova tabela: `bloqueios`
Precisa de migration para criar:
```sql
CREATE TABLE bloqueios (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  profissional_id UUID NOT NULL REFERENCES profissionais(id) ON DELETE CASCADE,
  salao_id UUID NOT NULL REFERENCES saloes(id) ON DELETE CASCADE,
  data_hora_inicio TIMESTAMPTZ NOT NULL,
  data_hora_fim TIMESTAMPTZ NOT NULL,
  dia_inteiro BOOLEAN NOT NULL DEFAULT false,
  motivo TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT chk_bloqueio_horario CHECK (data_hora_fim > data_hora_inicio)
);
```

### RLS
Mesmo padrão multi-tenant: `salao_id = public.get_salao_id()`

### Integração com conflito
A query de conflito do Story 2.3 deve incluir bloqueios:
```sql
-- Verificar bloqueio
SELECT id FROM bloqueios
WHERE profissional_id = $1
  AND data_hora_inicio < $new_fim
  AND data_hora_fim > $new_inicio
```

### Padrões
- Migration nova para tabela `bloqueios`
- Atualizar `Database` type em `types/supabase.ts`
- Server Actions: `createBloqueio`, `deleteBloqueio`
- Bloqueios renderizados como background events no FullCalendar
