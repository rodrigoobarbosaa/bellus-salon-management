# Story 9.1: Schema DB — Tabelas de Facturação Verifactu

**Status:** InProgress
**Epic:** EPIC-09 Verifactu
**Criado por:** River (@sm)
**Data:** 2026-03-30
**Depende de:** EPIC-05 (transacoes), EPIC-06 (configuracoes_fiscais)

---

## Story

**As a** proprietário de salão na Espanha,
**I want** que o sistema tenha uma estrutura de dados preparada para facturação eletrónica,
**so that** todas as faturas sejam armazenadas de forma inalterável e rastreável conforme a Ley Antifraude.

---

## Acceptance Criteria

1. Existe tabela `facturas` com campos: `id` (UUID), `salao_id` (FK), `serie` (TEXT), `numero` (INTEGER), `numero_completo` (TEXT GENERATED), `cliente_id` (FK nullable), `profissional_id` (FK nullable), `agendamento_id` (FK nullable), `fecha_emision` (TIMESTAMPTZ), `base_imponible` (NUMERIC), `iva_pct` (NUMERIC), `iva_valor` (NUMERIC), `irpf_pct` (NUMERIC), `irpf_valor` (NUMERIC), `total` (NUMERIC), `forma_pagamento` (ENUM), `hash_anterior` (TEXT), `hash_actual` (TEXT), `firma_digital` (TEXT nullable), `qr_data` (TEXT nullable), `estado_aeat` (ENUM: pendiente, enviado, aceptado, rechazado), `xml_verifactu` (TEXT nullable), `notas` (TEXT nullable), `created_at` (TIMESTAMPTZ)
2. Tabela `facturas` é append-only — RLS policy bloqueia UPDATE e DELETE
3. Campo `numero_completo` é GENERATED ALWAYS AS (`serie || '-' || lpad(numero::text, 6, '0')`)
4. UNIQUE constraint em `(salao_id, serie, numero)` garante unicidade por salão
5. Existe tabela `factura_lineas` com: `id`, `factura_id` (FK), `servico_id` (FK nullable), `descripcion` (TEXT), `cantidad` (INTEGER DEFAULT 1), `precio_unitario` (NUMERIC), `iva_pct` (NUMERIC), `subtotal` (NUMERIC), `created_at`
6. Existe tabela `factura_eventos` (audit log) com: `id`, `factura_id` (FK nullable), `salao_id` (FK), `tipo_evento` (ENUM: emision, anulacion, rectificacion, envio_aeat, error_aeat, consulta), `detalle` (JSONB), `usuario_id` (FK nullable), `created_at`
7. Tabela `factura_eventos` é append-only (sem UPDATE/DELETE)
8. Existe tabela `factura_envios_aeat` com: `id`, `factura_id` (FK), `xml_enviado` (TEXT), `response_code` (TEXT), `response_body` (TEXT), `status` (ENUM: pendiente, enviado, aceptado, rechazado, error), `enviado_em` (TIMESTAMPTZ nullable), `created_at`
9. RLS habilitado em todas as tabelas com políticas baseadas em `salao_id`
10. Índices criados: `(salao_id, serie, numero)`, `(salao_id, fecha_emision)`, `(salao_id, estado_aeat)`, `(factura_id)` em lineas/eventos/envios
11. Tipo TypeScript gerado/atualizado em `src/types/supabase.ts`
12. Migration SQL passa sem erros no `supabase db push`
13. `npm run build` sem erros
