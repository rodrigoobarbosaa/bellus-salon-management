# Story 3.2 — Fluxo de Reserva Online

**Epic:** EPIC-03 Portal Público & Notificações
**Status:** Done
**Prioridade:** MUST HAVE
**Depende de:** Story 3.1
**PRD Refs:** FR-02.3, FR-02.4, FR-02.5

---

## Story

Como **cliente**, quero **selecionar serviço, profissional, horário e confirmar minha reserva** pelo portal público, para que possa agendar sem ligar para o salão.

---

## Acceptance Criteria

- [x] AC-01: Step 1 — Cliente seleciona serviço (lista com nome, duração, preço)
- [x] AC-02: Step 2 — Cliente seleciona profissional (apenas os que atendem o serviço)
- [x] AC-03: Step 3 — Cliente seleciona data/hora (apenas slots disponíveis, respeita horário do salão e bloqueios)
- [x] AC-04: Step 4 — Cliente preenche nome, telefone (obrigatórios) e email (opcional)
- [x] AC-05: Confirmação mostra resumo: serviço, profissional, data/hora, preço
- [x] AC-06: Após confirmar, agendamento é criado com status "pendente"
- [x] AC-07: Detecção de conflito de horário — **BLOQUEIO RÍGIDO**: slots com agendamento existente NÃO aparecem como disponíveis (diferente do dashboard do profissional que permite override com confirmação — ver Story 2.3 regra de conflito dual)
- [x] AC-08: Interface em 4 idiomas (botão de troca de idioma)

---

## Dev Notes

- Wizard multi-step com state local (useState)
- Calcular slots disponíveis: pegar horário do salão, subtrair agendamentos e bloqueios existentes
- **IMPORTANTE — Regra de Conflito Dual (ver Story 2.3):**
  - Portal público (cliente): **BLOQUEIO RÍGIDO** — slot ocupado nunca aparece como disponível
  - Dashboard (profissional): permite override com confirmação
  - Na prática: a query de slots disponíveis deve EXCLUIR todos os horários com agendamentos existentes
- Service client para todas queries (sem auth)
- Reusar lógica de conflito do agendamentos.ts (modo `strict` para portal público)
- Rate limiting básico: max 5 reservas por IP/telefone por hora (implementar depois)
